**Application navigation behavior (from QuestionnaireForm.tsx):**
- `sections` array defines the 16-section order
- `handleNext()` increments `currentSectionIndex` (shown when not on last section)
- `handlePrevious()` decrements `currentSectionIndex` (hidden on first section via `invisible` class)
- On section `mental-health` (second-to-last), the forward button says "Generate Report" instead of "Continue"
- On `report` (last section), no forward button at all
- `Progress` component shows `(currentSectionIndex / (sections.length - 1)) * 100`%
- Step indicator shows "Step {currentSectionIndex + 1} of {sections.length}"
- `isContinueDisabled` returns true on `intro` if disclaimer unchecked, on `demographics` if required fields missing
- Dev page sidebar lists all sections with index numbers, clicking navigates via `window.location.href`

**Key constraints:**
- Tests against `http://localhost:3000` — app must be running
- Use `/dev` route (pre-fills mock data) for most tests
- Use `cy.navigateToSection()` for section jumping
- Import helpers from `../support/test-data` and `../support/assertions`

  <action>
Create `cypress/e2e/navigation.cy.ts` with the following test groups:

**describe('Questionnaire Navigation')**

**describe('Continue and Previous buttons')**
- `it('should show Continue button on the first section')` — visit `/dev?section=intro`, verify Continue button exists and is visible.
- `it('should not show Previous button on the first section')` — on intro, verify Previous button has `invisible` class or is not actionable.
- `it('should navigate to next section when Continue is clicked')` — on intro (mock data pre-fills disclaimer), click Continue, verify demographics section title appears.
- `it('should navigate back when Previous is clicked')` — navigate to demographics, click Previous, verify intro section title appears.
- `it('should show Generate Report button on second-to-last section')` — navigate to `mental-health` section, verify "Generate Report" button exists instead of "Continue".
- `it('should not show Continue or Previous on the report section')` — navigate to `report`, verify no Continue button (report is the last section). Previous should still be visible on report.

**describe('Progress bar')**
- `it('should show 0% progress on intro')` — navigate to intro, verify progress bar `aria-valuenow` is `0`.
- `it('should update progress when navigating forward')` — navigate to demographics (index 1), verify progress bar `aria-valuenow` is approximately `Math.round((1 / 15) * 100)` = 7. Use `closeTo` or check the rounded value.
- `it('should show 100% on report section')` — navigate to report, verify progress bar `aria-valuenow` is `100`.
- `it('should show correct step indicator')` — navigate to `daytime` (index 3), verify text contains "Step 4 of 16".

Import `SECTIONS`, `SECTION_TITLES`, `TOTAL_SECTIONS` from `../support/test-data` and assertion helpers from `../support/assertions`.

Use `cy.navigateToSection()` to set up each test's starting section. Use the assertion helpers wherever applicable (e.g., `assertSectionVisible`, `assertProgressBar`, `assertStepIndicator`).

For clicking Continue/Previous, use `cy.contains('button', 'Continue').click()` and `cy.contains('button', 'Previous').click()` — these match the actual button text in the UI.

Handle the intro section's `isContinueDisabled` — the mock data pre-fills `acceptedDisclaimer: true`, so Continue should be enabled on the dev route.
